1. What the app is

Horizon is a full-lifecycle retirement simulator for a two-person Ontario household. It models accumulation and decumulation with federal + Ontario tax, registered accounts (RRSP/RRIF, LIRA/LIF, DC), TFSA, unregistered accounts, CPP, OAS, optional DB, and a tax-aware “meltdown” strategy compared to a naive no-top-up baseline. Market risk is explored via seeded Monte Carlo.

Primary questions it answers

  • Can this lifestyle be funded through the plan horizon on the expected path?
  • Where does the money come from each year (which person, which account)?
  • What top-up / TFSA policy improves total tax and real estate vs never melting down?
  • Under random markets, what fraction of paths never miss spending?
Not goals: global multi-decade dynamic-programming tax optimum, multi-user backend, full CRA form fidelity, live ticker portfolios, or non-Ontario provinces.

Outputs are planning estimates, not advice. Policy numbers are pinned for tax year 2026 with sources in constants-2026.ts.

2. Architecture

Layered design: pure TypeScript engine + React shell. Long jobs can run in a Web Worker with progress and cancel. All data stays on-device (localStorage); there is no server.

On first open, the sample couple (Alex / Jordan) is typically analyzed automatically. Use Run full plan after edits; Market stress test runs Monte Carlo.

Simulation years are full calendar years from January 1. Default start year is the next Jan 1 if you open the app after January 1 (lib/defaultStartYear.ts).

4. Engine core

4.1 Policy constants

constants-2026.ts pins every statutory number with source retrievedOn indexation (cpi | frozen | wage. buildYearPolicy(cpiIndex) produces year-scaled brackets and limits. Footer / export expose POLICY_BASELINE (tax year 2026).

Major groups: FEDERAL, ONTARIO, LIMITS (TFSA/RRSP), CPP, OAS, RRIF minimum factors, Ontario LIF max factors, ECON_DEFAULTS (MC trials, vol, correlation), VALIDATION_ANCHORS.

Annual refresh checklist: docs/ANNUAL-POLICY-REFRESH.md.

4.2 Tax

computeTax(personIncome, splitIn, policy) computes federal + Ontario tax, surtax, Ontario health premium (OHP), OAS recovery, and non-refundable credits at the lowest rate. optimizeHouseholdTax / split helpers search pension-income split used inside the year loop.

4.3 Simulation conventions

  • Age in year t: t − birthYear (age attained Dec 31).
  • RRIF minimums: age at Jan 1.
  • Ontario LIF maximum: age attained (FSRA PE0196INF keying).
  • Salary: earned while ageDec31 < retirementAge.
  • Growth: mid-year convention B1 = B0·(1+r) + netFlow·(1+r/2).
  • Unregistered distributions: on opening balance; reinvested; raise ACB; taxable whether or not spent.
  • Spending solver: starts the first year either spouse is retired.
  • solverQuality: thorough (deterministic + tuner) vs fast (Monte Carlo, small tax noise).

5. Tax strategy pipeline

“Run full plan” does not claim a global tax optimum. It runs a lexicographic search: maximize funded years, then real after-tax estate. Tax savings are a derived metric, not the sole objective. 

analyzePlan(household)
  ├─ validateHousehold
  ├─ prepareForAnalysis (clone, stripPhantomDb, thorough)
  ├─ resolve TFSA level → default l4, reserve years → 2
  ├─ tuneStrategyBanded
  │    ├─ flat C grid (0 … max, coarse then fine)  [mc.tuneStrategy]
  │    ├─ age bands ≤71 / ≤80 / ≤120 (coordinate descent)
  │    └─ OAS soft-cap on (min C×CPI, OAS threshold)
  ├─ personCeilings = split flat C by registered balances
  ├─ topUpPriority = higherReg; tfsaAwareMeltdown = true
  ├─ re-simulate with person policy
  ├─ L4: tuneTfsaShare (TFSA-first share grid)
  └─ metrics vs naive C=0

Naive vs tax-aware

  • Naive: no deliberate meltdown (C = 0).
  • Tax-aware: banded/person C, TFSA policy L1–L4, surplus → TFSA room-first.
  • Tax saved: Δ(living tax + estate tax), not living tax alone.
  • Estate tax: gross balances − afterTaxEstate (simplified death calc via estateTaxOf).
  • Estate gain (real): primary real estate − naive real estate.

TFSA levels

LevelBehaviour
legacyTFSA last (classic order)
l1Taxable sources up to C, then TFSA, then overflow
l2L1 + OAS-aware income cap
l3L2 + TFSA spending reserve (N × spend)
l4 defaultL3 + searched TFSA-first share of discretionary W

Person-level policy

  • personCeilingsToday — split of flat C by RRSP/LIRA/DC weights (25% floor each when both material)
  • topUpPriority — who fills under their ceiling first (higherReg default)
  • tfsaAwareMeltdown — scale top-ups so ~65% after-tax proceeds fit remaining TFSA room
  • Surplus parking — spouse with more TFSA room first; residual unreg by unreg balance share
Agent trap: always merge the strategy object. Replacing it with a single key (e.g. only topUpCeilingToday) wipes TFSA level, bands, and person ceilings.

6. Year loop (simulate.ts)

Each calendar year roughly:

  1. Advance CPI; build YearPolicy (cached by CPI)
  2. RRIF/LIF conversions; TFSA/RRSP room accrual
  3. Income streams (salary, CPP, OAS, DB), forced mins, unregistered distributions
  4. Optional payroll (CPP/EI) and rough GIS
  5. Spending target: phases + one-time goals × survivor step-down × CPI
  6. Optional home sale → unregistered
  7. Solve discretionary withdrawals for lifestyle (bisection + pension split)
  8. allocateDiscretionaryW per TFSA level
  9. Person-aware RRSP top-ups under C / bands / person ceilings; TFSA-aware scale
  10. Park surplus: TFSA room-first, then unregistered
  11. Mid-year growth convention on accounts
  12. Optional first death at year-end (rollover, CPP boost)

Optional SimPath injects per-year inflation and account returns (Monte Carlo). Zero-volatility paths are bit-identical to the deterministic path.

7. Monte Carlo

  • Returns: lognormal with user’s µ as arithmetic mean; gross = exp(ln(1+µ) − σ²/2 + σ·z); default σ ≈ 11%
  • Correlation: shared market factor → pairwise ρ ≈ 0.85 across accounts/spouses
  • Inflation: fixed or AR(1) (UI toggle under Assumptions)
  • RNG: mulberry32 + deriveSeed(seed, trial)
  • Strategy pins: C, bands, OAS soft-cap, L4 share from analysis (prepareMonteCarloRun / householdForMonteCarlo)
  • solverQuality: fast
  • Success: share of trials with no spending shortfall year
  • UI estate: median (p50), worse (p10), better (p90) real after-tax estate

Median under volatility is usually below the smooth expected-return path (volatility drag + sequence risk). That is expected, not a bug.

13. Known simplifications

  • Estate tax uses unsplit final-year incomes (slight understatement)
  • Pension split is a fine grid, not closed-form global optimum; MC uses faster solver
  • Credits at lowest rate; no unused credit transfer between spouses
  • Payroll and GIS are optional rough approximations
  • Wage-indexed limits use CPI in v1
  • Bracket tables show taxableIncomePreSplit
  • TFSA-aware meltdown uses a fixed after-tax fraction (~65%) when scaling top-ups
  • Default volatility / correlation are modelling choices, not CRA data
  • Not a global multi-decade DP tax optimum

See also design doc and README “Known approximations”.

14. Notes for future agents

  • Parse docs/codebase-guide.json as the structured source of truth; this HTML is the human narrative.
  • Prefer pure helpers under src/lib/; only extend the year loop in simulate.ts when state must live there.
  • Strategy search objective stays funded-years → estate unless product explicitly changes.
  • Always merge strategy objects — never replace with a single key.
  • stripPhantomDb: zero DB entitlement must clear the whole db object (hidden accrual bug class).
  • Worker payloads must be structured-cloneable; do not put functions in postMessage payloads.
  • Policy updates: change constants-2026.ts and docs/ANNUAL-POLICY-REFRESH.md together.
  • Person A accent = lime; person B = magenta (drawdown / charts).
  • After engine changes: npm test; consider fuzz + engine-ui-path.
  • Do not invent CRA rates — pin them as Sourced<> with retrievedOn.
